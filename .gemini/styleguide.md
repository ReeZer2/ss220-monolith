## 4. Правила анализа C#-кода (SS14/SS220 Specific)

### 4.1. Стиль и Нейминг (C# Guidelines)

* **PascalCase:** Классы, Методы, Свойства, Публичные поля (`PublicMethod`, `SystemName`).
* **camelCase:** Локальные переменные, приватные поля (`_dependency`, `targetEntity`).
* **Префиксы:** Приватные поля должны начинаться с `_` (например, `_entityManager`).
* **Интерфейсы:** Должны начинаться с `I` (`IEntityManager`).
* **Пространства имен:** Проверять соответствие структуре папок (напр., `Content.Server.Fluids`).
* **Implicit Typing (`var`):**  Использовать `var`, когда тип очевиден из правой части (`new List<int>()`).

* **Allman Style:**  В C# принято переносить открывающую скобку `{` на **новую строку** для классов, методов и управляющих конструкций (`if`, `for`, `foreach`).
* **Опускание скобок:** Допускается опускать скобки для **однострочных** `if/else`, если это улучшает читаемость и не создает двусмысленности.
* **Правило "Одной строки":** Между методами, свойствами и логическими блоками должна быть **ровно одна** пустая строка. Две и более — это лишняя пустота.
* **Запрет на "padding":** Запрещено оставлять пустые строки сразу после открывающей скобки `{` или перед закрывающей `}`.
* **Смыслу место:** Если код разделён строчками, то он должен иметь разный смысл, например инициализация потоков может быть отделена пустой строкой от операций с ними, например, записью в них или сжиманием их.

### 4.2. Современный стиль SS14

* **Обработчики событий**: Считать устаревшими обработчики с такой сигнатурой `OnEvent(EntityUid uid, MyComponent comp, Event args)`, вместо них нужно использовать `OnEvent(Entity<MyComponent> ent, ref Event args)`
* **Надёжность:** Вместо метода `Method(EntityUid uid, MyComponent comp)` предпочитать такой вариант `Method(Entity<MyComponent> entity)`, потому что в нём debug сборка автоматически проверит принадлежность компонента к переданному entity.
* **Ясность:** Вместо string как аргумент и как константу передающий или хранящей прототип (наследники `Prototype` и помеченные атрибутов `[prototype]`, а так же то что потому будет использовано в `_prototype.Resolve` или `_prototype.TryIndex`) использовать `ProtoId<T>`, что позволяет в debug сборке проверять корректность.
* **События:** События всегда следует по возможности представлять в виде `record struct`, потому что они лучше для производительности, чем классы. Например
`[ByRefEvent]
public record struct SomeAttemptEvent(bool Cancelled=false);`

* **Удобное API**: для публичных методов систем, которые что-то меняют в компонентах или получают какие-то характеристики предпочитать методы `TryDoSomething` или `TryGetSomething` или `ResolveSomething`, учитывая что `Try...` должен возвращать bool строго требующий быть true при успешном действии, а методы Resolve должны обрабатывать **каждый** false сообщая об ошибке через логгер (например `Log.Error($"Got invalid entity {ToPrettyString(uid)}!")`)
* **Знание меры:** В настоящее время `[DataField]` включает `[ViewVariables(VVAccess.ReadWrite)]`, поэтому вместо `[DataField, ViewVariables(VVAccess.ReadWrite)]` просто пишется `[DataField]`
* **Плавающие биты:** Не нужно обновлять поле данных `NextTimeWork` таким образом: `comp.NextTimeWork = curTime + comp.UpdateInterval;`, что приводит к ошибке округления чисел с плавающей точкой. Предпочитать нужно просто вариант `comp.NextSound += comp.UpdateInterval;`. В качестве временной метки всегда используйте `TimeSpan`, а не накапливайте время в виде числа `float`.


### 4.3. Игровые требования при работе с кодом

* **Имена:** При использовании имён в видимых игрокам методах (например `_popup.PopupEntity`), нужно использовать Identity.Name(uid) вместо этого, причина - корректно отображение сокрытия личности, например «молодой человек» вместо УристМакЧеловек.

### 4.4. Архитектура ECS (Entity Component System)

* **Закон Разделения:**
* ❌ **Плохо:** Логика внутри класса Компонента (`MyComponent.cs`).
* ✅ **Хорошо:** Компонент — это только данные. Вся логика — в Системе (`MySystem.cs`).
* **Dependency Injection:** Настаивать на использовании атрибута `[Dependency]` для внедрения зависимостей, вместо `IoCManager.Resolve<T>()`, где это возможно.
* **UI *.xaml.cs (Важно):** Возможно использование `IEntityManager` как способ получения системы
* **Events (События):** Использовать `SubscribeLocalEvent<T>`

### 4.5. Производительность и Память

* **Hot Paths (Update/FrameUpdate):**
* Избегать LINQ (`Where`, `Select`, `ToList`), создающего аллокации (замыкания, итераторы) в методах, вызываемых каждый кадр. Для простых проверок используйте циклы `for/foreach`.
* Запрет на `new List<T>()` или другие аллокации коллекций в цикле обновления. Использовать пулинг или переиспользование коллекций.

* **Entity Lookup:**
* ❌ **Плохо:** Перебор всех сущностей в мире (`EntityManager.GetEntities()`).
* ✅ **Хорошо:** Использование `EntityQuery<T>`, фильтров или подписка на компоненты.

* **Resolving:** Избегать `Resolve<T>` внутри циклов. Получать компоненты заранее или использовать `GetEntityQuery`.

### 4.6. Стабильность и Безопасность

* **Валидация Сущностей:** Проверка `Deleted(entity)` и `!entity.Initialized` перед манипуляциями, если работа идет с отложенными таймерами.
* **Null-Safety:** C# 9.0+ паттерны. Использовать `TryGetComponent` вместо `GetComponent` (который кидает исключение), если наличие компонента не гарантировано на 100%.

### 4.7. Локализация (Loc)

* **Хардкод строк:** Запрещен для любого текста, видимого игроку, исключение - текст для администрации в чат админов серверов. Исключение - текст в ентити на игровых картах.
* **Loc.GetString:** Весь текст должен проходить через `Loc.GetString("string-id")`.
* **Форматирование:** Использовать аргументы локализации (`Loc.GetString("my-message", ("user", uid))`) вместо конкатенации строк, там где это возможно.

### 4.8. Правила YAML (Прототипы и Конфиги)

* **Компактность списков (Compact Lists):**
* Мы используем компактный стиль списков. Дефис `-` элемента списка **НЕ должен** иметь дополнительного отступа относительно родительского ключа.
* ❌ **Плохо (Лишний воздух):**
```yaml
effects:
  - !type:DoAfter
  - !type:Popup

```

* ✅ **Хорошо (Плотная структура):**
```yaml
effects:
- !type:DoAfter
- !type:Popup

```

### 4.9. Маркировка Изменений

Если Творец вносит изменения в файлы, которые **НЕ** лежат внутри папок с именем `_Exodus` (например, модифицирует файлы апстрима), он обязан оставить метку. Это позволяет отличить оригинальный код от модификаций форка.

* **Однострочное изменение:** Должно сопровождаться комментарием в конце строки: `// Exodus-feature-name` (для C#) или `# Exodus-feature-name` (для YAML).
* **Блок изменений:** Должен быть обернут в открывающий и закрывающий комментарии:
```csharp
// Exodus-feature-name-begin
... ваш код ...
// Exodus-feature-name-end

```
* **Формат:** Не строгий, но рекомендуемый: `// Exodus My feature` или блоки `// Exodus-begin` ... `// Exodus-end`. Главное — наличие метки.
* **Разделение изменений:** Хороший тон - большие изменения в файлах апстрима, например `ExampleUpstreamSystem.cs`, переносить в новый файл `ExampleUpstreamSystem.Exodus.cs` помечая родительский класс `partial`
* **Незнание - тьма:** Описанные изменения должны соответствовать написанному коду
* **Знание - свет:** Изменения, считающиеся исправлениями ошибок, затрагивающие опциональные игровые механики/интеракции, нужно добавлять в чейнджлог CL, чтобы сообщать самую актуальную информацию
